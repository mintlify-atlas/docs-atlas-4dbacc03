TRC requires a MySQL 8+ or MariaDB 10.4+ database named `sistema_viajes`. The repository includes a complete SQL dump at `BD/sistema_viajes.sql` that creates all tables, indexes, foreign keys, and loads sample data.

<Note>
  The SQL dump was generated with **MariaDB 10.4** via phpMyAdmin 5.2.1. It is fully compatible with MySQL 8 and with MariaDB 10.4 or later.
</Note>

## Setup steps

<Steps>
  <Step title="Start the MySQL server">
    Make sure your MySQL or MariaDB service is running before proceeding.

    ```bash
    # Linux (systemd)
    sudo systemctl start mysql

    # macOS (Homebrew)
    brew services start mysql
    ```
  </Step>

  <Step title="Create the database">
    Connect to MySQL and create the `sistema_viajes` database.

    ```sql
    CREATE DATABASE sistema_viajes;
    ```

    Or do it directly from the shell:

    ```bash
    mysql -u root -e "CREATE DATABASE sistema_viajes;"
    ```

    <Note>
      If your MySQL root user has a password, add the `-p` flag and enter it when prompted: `mysql -u root -p`.
    </Note>
  </Step>

  <Step title="Import the SQL dump">
    From the root of the TRC project directory, run:

    ```bash
    mysql -u root sistema_viajes < BD/sistema_viajes.sql
    ```

    This single command creates every table, adds indexes and foreign key constraints, and inserts the sample rows.
  </Step>

  <Step title="Verify the tables">
    Connect to the database and list the tables to confirm the import succeeded.

    ```sql
    USE sistema_viajes;
    SHOW TABLES;
    ```

    Expected output:

    ```
    +---------------------------+
    | Tables_in_sistema_viajes  |
    +---------------------------+
    | colaborador_sucursal      |
    | colaboradores             |
    | rol                       |
    | sucursales                |
    | transportistas            |
    | usuarios                  |
    | viajes                    |
    +---------------------------+
    7 rows in set
    ```
  </Step>
</Steps>

## Tables created

The dump creates the following seven tables.

| Table | Primary key | Description |
|---|---|---|
| `colaboradores` | `cl_cedula` (VARCHAR 13) | Employee directory |
| `colaborador_sucursal` | `(cl_cedula, sc_id)` composite | Junction table linking employees to branches with a travel distance |
| `rol` | `rl_id` (INT, AUTO_INCREMENT) | User roles (Gerente de Tienda, Usuario) |
| `sucursales` | `sc_id` (INT, AUTO_INCREMENT) | Branch locations |
| `transportistas` | `tr_id` (INT, AUTO_INCREMENT) | Transport providers with a per-trip fare |
| `usuarios` | `us_id` (INT, AUTO_INCREMENT) | Application user accounts |
| `viajes` | `vj_id` (INT, AUTO_INCREMENT) | Trip records |

## Foreign key constraints

The schema enforces referential integrity with the following constraints. All use `ON UPDATE CASCADE`, meaning a primary key change propagates automatically.

| Constraint | Child table | Child column | Parent table | Parent column |
|---|---|---|---|---|
| `colaborador_sucursal_ibfk_1` | `colaborador_sucursal` | `cl_cedula` | `colaboradores` | `cl_cedula` |
| `colaborador_sucursal_ibfk_2` | `colaborador_sucursal` | `sc_id` | `sucursales` | `sc_id` |
| `usuarios_ibfk_1` | `usuarios` | `rl_id` | `rol` | `rl_id` |
| `viajes_ibfk_2` | `viajes` | `tr_id` | `transportistas` | `tr_id` |
| `viajes_ibfk_3` | `viajes` | `us_id` | `usuarios` | `us_id` |
| `viajes_ibfk_4` | `viajes` | `sc_id` | `sucursales` | `sc_id` |
| `viajes_ibfk_5` | `viajes` | `cl_cedula` | `colaboradores` | `cl_cedula` |

<Note>
  The `viajes` table references `cl_cedula` directly rather than going through `colaborador_sucursal`. The Sequelize model for `viajes` tracks the collaborator via the `cl_id` foreign key name in the application layer, but the SQL schema column is named `cl_cedula`.
</Note>
